## Current Code Baseline

- The active loader keeps current raster improvements and removes vector, bubble/frame, and text fallback rendering.

- The prior dirty branch state is preserved in git stash as `backup before returning to main and dropping vector-bubble-text work`.

## Native Loading Notes

- Current selected-tile decode state: CHNKExta selected-tile reads stop once the last requested tile block has been found, so unrelated trailing blocks are not scanned or inflated. Visible-region reads express the intersecting tile rectangle as a compact selection and stream matching blocks through a reusable scratch buffer into `TileRegionWriter`, avoiding per-region tile-index vectors, owned block lists, and block-ref vectors on the render path. Low-level external block readers live in `clip_file/src/external/reader.rs`.
- Decode/cache experiment state: `RIZUM_CLIP_DECODE_PROFILE=1` records selected raster/mask tile counts, skipped empty tile counts, compressed bytes, zlib inflate, RGBA swizzle, mask R8 decode/crop, atlas chunk construction, sparse atlas pool update, region patch render, and worker total timings in CLI debug output and Blender worker JSON. `RIZUM_CLIP_PARALLEL_TILE_DECODE=1` is an opt-in adaptive prototype for independent compressed CHNKExta tile inflation; it stays sequential below 16 selected compressed tiles or 1 MiB compressed data and caps worker threads at 8. Release medians on `Test_RealArt`, `Ref_Terra404_Live2D`, `Test_AddGlowMultiply`, and `Test_ToneCurve` showed identical pixels but no worker-total improvement, so decoding/zlib/atlas chunk construction is not the current full-render bottleneck and the parallel path should not be enabled by default.
- Render/cache diagnostics state: `RIZUM_CLIP_RENDER_PROFILE=1` records source selection, one-time GPU device init, render-program planning, tile-event payload buffer build, GPU pass encoding, queue submit/poll, CPU wait-proxy GPU execution, readback copy/CPU copy, patch payload extraction, checkpoint reconstruction, sparse atlas update, legacy barrier segment time/count, tile-local segment time/count, checkpoint cache hit/miss/store/evict counters, and top slow render segments in CLI debug output and Blender worker JSON. Worker JSON also includes a debug render task graph (`DecodeTile`, `UploadAtlasSlot`, `RunSegment`, `BuildCheckpoint`, `ReadbackPatch`, `RegionFallback`) and tile-cache/signature feasibility diagnostics when the env var is enabled. Release medians show CLI process startup/device init dominates small samples, while Terra/RealArt still spend meaningful time inside tile-local segment execution and legacy barrier segments; readback and source selection are not current full-render bottlenecks. Large scope atlas expansion above the current 8192 cap was probed and rejected: it removed `atlas_layout_unavailable` fallbacks but routed Terra/RealArt through less faithful large ThroughScope tile-local paths (`Ref_Terra404_Live2D` visible premul pixels worsened and `Test_RealArt` premul max rose to 4), so keep those fallbacks until focused legacy-vs-tile guards prove the large-scope path faithful.
- `clip_file::metadata` now presents typed `.clip` metadata records through a small re-export Interface while keeping SQLite/schema details inside focused reader modules. `metadata.rs` is wiring only; records live in `metadata/records.rs`, shared SQLite/schema helpers in `metadata/schema.rs`, and implementations in `metadata/summary.rs`, `metadata/layer_graph.rs`, `metadata/raster_source.rs`, `metadata/mask_source.rs`, `metadata/filter_source.rs`, and `metadata/paper_color.rs`.

The accepted Blender add-on path is a Rust renderer core with `wgpu` GPU compositing behind an explicit C ABI bridge. Do not reintroduce the Python compositor/loader or sidecar PNG workflow into the installable add-on. Do not maintain a duplicate native CPU compositor; the old project-root Python compositor has been removed, so use CSP PNG exports plus native `clip_cli --compare-png` as verification references.
- Current native integration milestone: the Blender image-datablock bridge is the accepted add-on runtime path. The add-on is branded as `Rizum Clip Reload`, while the import menu stays the concise legacy `File > Import > Clip Studio (.clip)` label; `clip_studio_importer/native_bridge.py` calls the packaged `clip_cli` worker so Rust/wgpu rendering runs outside Blender's UI process, reads the worker's native RGBA8 output, creates or updates generated Blender images, vertically flips rows while uploading into Blender's bottom-row-first `Image.pixels` storage, and stores source/render metadata including source mtime, source size, source SHA-256, renderer ABI, and renderer version. Initial import and manual reload both schedule the worker asynchronously: import does not create a placeholder image, and only creates/shows the generated Blender image in already-open Image Editor or UV Editor areas after the native worker returns real canvas pixels; the first import then schedules an initial pack as a separate main-thread timer. The background watcher uses lightweight mtime/size checks with a 0.1s minimum timer interval; `load_post` also checks SHA-256 so same-stat source replacements can refresh after reopening a `.blend`. Blender-relative source paths are resolved through `bpy.path.abspath` during reload/watch/load-post checks. If the `.clip` source is missing after reopening a `.blend`, the add-on keeps the packed pixels visible and records `missing_source`. - First OIIO milestone result: the plugin builds against Blender 5.0.1's bundled OpenImageIO 3.0.9.1 and opens through Blender's Python `OpenImageIO` module (`64x64`, RGBA, first pixel `[0,0,224,255]`). - Blender source probe conclusion: ImBuf iterates a static `IMB_FILE_TYPES` table. OIIO-backed formats such as PSD have explicit ImBuf entries that call `imb_oiio_check(..., "psd")` / `imb_oiio_read(...)`; Blender does not pass arbitrary unknown extensions to OIIO plugins. External OIIO plugin loading alone is therefore insufficient for stock Blender image datablock direct-load.
- Second milestone result: the stock Blender bridge works. The spike under `native/spikes/blender-image-bridge/` opens the OIIO placeholder plugin, creates a generated `bpy.types.Image`, and uploads RGBA bytes with `foreach_set` without writing a sidecar PNG. It is not a true file-backed image format; `Image.reload()` and external image auto-reload add-ons will not work unless the add-on owns reload and updates the datablock. Accepted persistence model: pack the latest rendered pixels into the `.blend` by default and store `.clip` source-tracking custom properties on the image. On `.blend` reopen, Blender shows the packed last render immediately; the add-on then checks source mtime/hash and re-renders through native code if the `.clip` changed. If the `.clip` is missing, keep packed pixels visible and report the missing source.
