## Current Scope

- In scope: flattened raster layers, folders, masks, clipping, blend modes, adjustment/filter layers, Blender import/reload behavior, and native generated-image loading.
- Out of scope: vector strokes/fills, bubble/frame renderers, text renderers, 3D layers, animation timelines, and write-back.
- Historical vector/native-material work is archived evidence only. Do not resume it unless Rizum explicitly reopens the scope.

## Current Code Baseline

- The active loader keeps current raster improvements and removes vector, bubble/frame, and text fallback rendering.
- Preserved raster paths include `expected_len` tile reads, render/mask offscreen offsets, grayscale/monochrome raster decode, filter-layer math, transparent RGB export convention, clipped-group handling, and `clip_base` allocation hygiene.
- The main compositor now renders into straight `uint8` RGBA buffers. Transparent caches are initialized as white RGB with alpha 0, matching CSP's ability to retain RGB under transparent alpha; the old premultiplied-float path remains only as a compatibility/debug fallback for callers that pass float buffers.
- Tracked vector, bubble/frame, text, thickness, native-material, and probe artifacts have been removed from the active tree.
- The prior dirty branch state is preserved in git stash as `backup before returning to main and dropping vector-bubble-text work`.

## Native Loading Notes

- Current selected-tile decode state: CHNKExta selected-tile reads stop once the last requested tile block has been found, so unrelated trailing blocks are not scanned or inflated. Visible-region reads express the intersecting tile rectangle as a compact selection and stream matching blocks through a reusable scratch buffer into `TileRegionWriter`, avoiding per-region tile-index vectors, owned block lists, and block-ref vectors on the render path. Low-level external block readers live in `clip_file/src/external/reader.rs`.

The accepted Blender add-on path is a Rust renderer core with `wgpu` GPU compositing behind an explicit C ABI bridge. Do not reintroduce the Python compositor/loader or sidecar PNG workflow into the installable add-on. Do not maintain a duplicate native CPU compositor; use the project-root Python loader and CSP PNG exports only as slow external references during development.
- Current native integration milestone: the Blender image-datablock bridge is the accepted add-on runtime path. `clip_studio_importer/native_bridge.py` calls the packaged `clip_cli` worker so Rust/wgpu rendering runs outside Blender's UI process, reads the worker's native RGBA8 output, creates or updates generated Blender images, vertically flips rows while uploading into Blender's bottom-row-first `Image.pixels` storage, and stores source/render metadata including source mtime, source size, source SHA-256, renderer ABI, and renderer version. Initial import and manual reload both schedule the worker asynchronously: import does not create a placeholder image, and only creates/shows the generated Blender image in already-open Image Editor or UV Editor areas after the native worker returns real canvas pixels; the first import then schedules an initial pack as a separate main-thread timer. Reload updates the existing image on the main thread and does not pack immediately; reload renders mark `clip_pack_status=needs_pack`. The Image Editor panel shows source, non-ready render status, bare pack status (`Packed` / `Needs Pack`) with adjacent `Pack`, errors, unsupported-node locators when present, `Manual Reload`, and a lower `Copy Diagnostic` action; render timing and `Open Diagnostics` are hidden unless Developer Mode is enabled. It intentionally hides native renderer mode, renderer version, full-support summaries, resource counts, largest raster/mask metadata, successful packaged-worker status, and save-time pack hints from the normal UI. A persistent `save_pre` handler packs dirty native images before saving the `.blend`. The add-on no longer exposes a user-facing native renderer override; preferences group `Autoreload .Clip` plus `Check Timer Frequency (s)` in a reload box and `Debug log` plus `Developer Mode` in a developer box, while only missing packaged-worker status is shown as an error. Native imports, manual reload, background watcher updates, and Blender `load_post` freshness scans go through the generated-image path without writing sidecar PNGs. The background watcher uses lightweight mtime/size checks with a 0.1s minimum timer interval; `load_post` also checks SHA-256 so same-stat source replacements can refresh after reopening a `.blend`. Blender-relative source paths are resolved through `bpy.path.abspath` during reload/watch/load-post checks. If the `.clip` source is missing after reopening a `.blend`, the add-on keeps the packed pixels visible and records `missing_source`. Render failures are stored on the image as `clip_reload_status=error` plus `clip_reload_error`, successful renders clear old errors, and the Image Editor panel shows readable non-ready status, missing-source, pack status, and error diagnostics while keeping renderer version in copied/opened diagnostics only. Render timing distinguishes end-to-end last render duration from phase timings stored on the image and shown in diagnostics: native worker, worker output read, RGBA8-to-Blender-float conversion, `foreach_set`, `image.update()`, `image.pack()`, and Blender upload total; deferred pack timing is additionally stored as `clip_pack_last_seconds`, and UI timing lines are Developer Mode-only. The Image Editor timing UI tolerates a stale in-process `native_bridge` module without timing constants, covering Blender hot-update/module-cache skew after reinstalling the add-on. The native worker/C ABI expose metadata-only native support summaries, and the add-on stores support status, unsupported counts, support details, and compact unsupported layer/node/kind issue locators (`clip_support_locations`) for diagnostics, but does not surface full-support/resource summaries in the normal panel. Large raster/mask byte totals remain native CLI/debug data only and are not written to Blender IDProperties or shown in the support diagnostics UI, avoiding Blender's 32-bit custom-property integer limit. Native support reports now include `.clip` layer names when `LayerName` is present, so Blender summaries and copied locators can show entries such as `layer 10 [Layer 1] node 2 Raster` instead of bare ids. The Python bridge parses unsupported details into structured layer records with layer id, optional layer name, node id, kind, and reason; when unsupported nodes exist, the panel previews the first few details, shows a locator summary, can expand to show the full stored list, copy unsupported layer locations, and open/copy focused diagnostics containing source path, size/hash, status, timing, canvas, renderer version, unsupported nodes, and errors. These locators are for issue reports and source-file follow-up only; the Blender add-on scope is flattened texture import/reload, not layer navigation. `tools/build_blender_addon.py` builds `clip_studio_importer.zip` with `__init__.py`, `native_bridge.py`, and the local release `clip_cli` worker plus `clip_capi` library under `clip_studio_importer/native/` for automatic discovery. The current installable add-on version is `0.8.63`. The installable package no longer includes `clip_studio_importer/clip_loader.py` or a `Use native renderer` off switch.
- Diff reload now uses persistent native worker state across the native worker and Blender bridge. `clip_runtime::reload_diff` builds a JSON-serializable reload manifest from the visible render nodes plus raster/mask CHNKExta compressed-tile fingerprints; `clip_cli --blender-render-rgba ... --blender-render-json ...` writes the manifest in `reload_diff.manifest`, and `--blender-reload-old-json <manifest.json>` plans `full`, `patch`, or `no_change` output. `clip_cli --blender-render-server` keeps one native process and one reusable `RuntimeGpuRenderer` alive across Blender requests, so reloads no longer pay per-request process startup and wgpu device initialization unless the persistent transport fails. Same-canvas/root/node-order raster/mask tile payload changes can return dirty rect payloads; unchanged manifests skip rendering/upload payloads; canvas/root/node-order or non-raster semantic changes conservatively full-update. Persistent server renders keep a raster/mask GPU texture cache keyed by source metadata plus CHNKExta compressed-tile fingerprints; patch reloads render the full semantic stack into dirty-region GPU targets and read back only those patch targets instead of rendering the full canvas and slicing afterward. This reuses unchanged input GPU textures and the previous Blender image for unchanged output regions; fuller per-subtree tile-DAG cache invalidation remains a later renderer milestone, not a CPU fallback or post-processing path. `native_bridge.py` stores `clip_reload_manifest_json` on the Image, passes it on reload, and applies patch rows into `Image.pixels` using Blender's bottom-row-first coordinates. Normal UI hides the diff details; copied/opened diagnostics include reload diff mode/patch count. The current installable add-on version is `0.8.63`.
- Native rewrite architecture is documented in `docs/native-code-architecture.md`. The initial Rust workspace lives under `native/rust/` with separate crates for `clip_model`, `clip_file`, `clip_graph`, `clip_gpu`, `clip_runtime`, `clip_capi`, and `clip_cli`; root files are wiring only and must not accumulate renderer behavior. `clip_capi/src/lib.rs` is back under the 500-line split threshold after moving support report text formatting and tests into `clip_capi/src/support_report.rs`.
- First OIIO milestone source lives under `native/oiio/`. It is a minimal C++20 `ImageInput` plugin that registers `.clip`, validates the `CSFCHUNK` magic, and returns a deterministic 64x64 RGBA placeholder image. This is only format plumbing; it does not parse layers or call Rust yet.
- First OIIO milestone result: the plugin builds against Blender 5.0.1's bundled OpenImageIO 3.0.9.1 and opens through Blender's Python `OpenImageIO` module (`64x64`, RGBA, first pixel `[0,0,224,255]`). Blender's ordinary `bpy.data.images.load()` and `bpy.ops.image.open()` still do not load `.clip`; they create an empty image and log `IMB_load_image_from_memory: unknown file-format`.
- Blender source probe conclusion: ImBuf iterates a static `IMB_FILE_TYPES` table. OIIO-backed formats such as PSD have explicit ImBuf entries that call `imb_oiio_check(..., "psd")` / `imb_oiio_read(...)`; Blender does not pass arbitrary unknown extensions to OIIO plugins. External OIIO plugin loading alone is therefore insufficient for stock Blender image datablock direct-load.

The spike under `native/spikes/blender-image-bridge/` opens the OIIO placeholder plugin, creates a generated `bpy.types.Image`, and uploads RGBA bytes with `foreach_set` without writing a sidecar PNG. It is not a true file-backed image format; `Image.reload()` and external image auto-reload add-ons will not work unless the add-on owns reload and updates the datablock. Accepted persistence model: pack the latest rendered pixels into the `.blend` by default and store `.clip` source-tracking custom properties on the image. On `.blend` reopen, Blender shows the packed last render immediately; the add-on then checks source mtime/hash and re-renders through native code if the `.clip` changed.
